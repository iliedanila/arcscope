arcscope — architecture-aware code-navigation MCP

A fully-local MCP server that gives an AI coding agent three stacked views of an unfamiliar codebase — and a repo-declared architecture vocabulary answered live against current code.
Status: Draft spec — pre-build Date: 2026-06-19 Author: Ilie + Claude Seeds: a new standalone repo (this file travels with it)

1Thesis & the wedge

An AI agent dropped into a codebase re-derives its structure with grep every single session. arcscope captures that structure once and answers it on demand — and crucially, it lets a repo name its own architecture and binds each named concept to a predicate that is re-run against the current tree, so the answer can never silently rot.

The tool sits above the language level. It does not reinvent symbol navigation — that is a solved problem (tree-sitter for breadth, language servers for precision). It stacks three views and earns its keep on the third:

  1. Symbols — where a thing is defined, who references it.
  2. The module / dependency graph — what depends on what, what the hubs are.
  3. The architecture vocabulary — the repo’s own concepts (“the repository tokens”, “the action pipeline”, “the plugin registry”), each answered live.

The wedge, stated against the closest competitor. Serena (the nearest prior art) already wraps language servers behind MCP and writes project “memories”. But those memories are static prose — hand-written once, never checked against code, stale the moment someone moves a file, and the agent has no way to know. arcscope’s vocabulary entries are executable locators recomputed on every query. An answer is never cached prose; drift becomes a signal instead of a surprise. That live-verified, code-bound semantics is the one thing none of the prior art offers — and it is where all net-new effort goes.

2The problem — four pains

These are the four navigation pains the agent hits today, why grep/embeddings fall short, and which layer answers each.

#PainWhy grep / RAG falls shortAnswered by
1Find defs / refs — where is X defined, who calls itText grep on a name yields false positives (comments, shadowing, same-named symbols) and false negatives (re-exports through barrels); no precision signalEngine
2Architecture flow — which files participate in a named pipeline/patternNo primitive answers a concept-level query; the agent reconstructs the flow by reading file after fileKnowledge
3Module / dep graph — what depends on this, what are the hubsgrep can’t traverse edges; following imports by hand is slow and lossyGraph
4Re-learning conventions every sessionConventions live in CLAUDE.md prose / tribal knowledge; re-derived each conversation, with no way to tell which are still trueKnowledge

Pains 1 and 3 are solved by borrowing engines. Pains 2 and 4 are genuinely unserved — and solving them is the product.

3Prior art & where we fit

We surveyed the field so we don’t reinvent it. The summary: everything good at symbol/graph nav has no notion of declared architecture; the one tool with “knowledge” (Serena) keeps it as unverified prose.

4Principles

  1. Borrow the engine, build the vocabulary. Never reinvent symbol nav. tree-sitter is the always-on breadth substrate; LSP is a future optional precision overlay. All net-new effort goes into the vocabulary layer.
  2. Live over stale. Every vocabulary answer is recomputed against the current tree at query time. We never serve cached prose. Drift (a locator whose resolved set diverges from its accepted anchors) is surfaced as a signal.
  3. Nothing leaves the machine. No network calls, no embeddings API, no telemetry — at query time and at server spawn (see §8). A hard guarantee, not a toggle.
  4. General signals, not build-tool bindings. Module grouping comes from universal signals (directory structure, import-graph clustering, tsconfig paths, workspace manifests) — never nx.json, BUILD files, or any one tool’s project model.
  5. Earn adoption; assume the agent will grep instead. A registered-but-ignored tool is worthless. Adoption is an empirical bet validated with a kill-criterion in Phase 0 — not a copywriting exercise.
  6. Thin vertical slice before breadth. Ship one pain end-to-end before widening. Nothing speculative; nothing built before a measurement justifies it.

Two host couplings stated honestly up front: arcscope analyzes any language, but it runs on Node ≥ 20 (web-tree-sitter + the MCP SDK) — a polyglot shop with no Node toolchain is a documented prerequisite, not a silent dead end. And “fully local” is a guarantee about behavior, not a claim of zero native code: the eventual persistence layer may use a native module (see Open decisions).

5Architecture — three stacked layers

A single local Node/TypeScript process speaking MCP over stdio. Three layers, borrowed at the bottom, built at the top.

Engine — breadth, always on

Parse every file with web-tree-sitter (WASM grammars loaded lazily per file extension — no per-arch native build, which is the only practical way to be “any machine, any language, cold”). Extract definitions and import edges via each grammar’s tags.scm query with one generic query-runner. v1 is tree-sitter only. Every result is labeled with a precision tier so the agent calibrates trust; an LSP overlay is a future tier, not a v1 dependency.

Graph — derived, not borrowed

The dependency graph is a view over the engine’s import edges, not a separate subsystem. Files are nodes; resolved imports are edges. Module grouping is inferred with a deterministic precedence, but with one honesty correction from review (see §11): the directory + import-graph clustering fallback is the primary path, because the JS-only signals above it are frequently empty — including on this very repo, which has no package.json workspaces and whose real boundaries live in Nx files we deliberately refuse to read. tsconfig paths happen to map cleanly to libs/* here, so they’re a useful secondary signal.

Knowledge — the architecture vocabulary (the product)

A repo-committed manifest declares named concepts, each bound to a locator resolved through arcscope’s own engine. arch_query re-runs the locator against current code and returns live locations plus a freshness flag — never stored prose. Detail in §7.

6Tool surface

Minimal and sharp — one or two tools per pain, named specifically (generic names like query kill discovery). Phase tags show when each ships.

ToolShipsPurposeOutput (sketch)
find_defP0Pain 1 — where a symbol is defined, with outline. Definitions are the easy, honest half: tree-sitter beats grep here without any cross-module resolution.[{symbol, kind, file, line, signature, precisionTier}]
find_refsP1Pain 1 — who references a symbol. Deliberately not in P0: to actually beat grep it must follow tsconfig aliases + barrel re-exports, which needs the import-edge index built in P1 (see §11).[{file, line, snippet, refKind, precisionTier}]
dep_graphP1Pain 3 — the module/dep graph or a focused neighborhood.{nodes, edges, groups}, token-bounded
arch_listP2Pain 4 — list the repo’s declared concepts (progressive disclosure: names first).[{concept, id, description, freshness, instanceCount}]
arch_queryP2Pains 2 & 4 — resolve one named concept live against current code.{concept, resolved:[{file,line,stage,snippet}], freshness, drift?}

The precision-tier field ships from P0 as a constant "tree-sitter"; it exists so the surface has room for an LSP tier later without changing tool shapes — and so the agent is never told a heuristic answer is compiler-accurate.

7The architecture vocabulary — the differentiator

A concept bound to an executable locator that is recomputed on demand is precisely what no competitor offers: Serena names knowledge but never verifies it; ast-grep verifies but never names or registers; embeddings do neither. Binding name → predicate → live answer makes the manifest captured-once, queried-forever, and structurally unable to silently rot.

Locator model (v1)

A locator resolves through arcscope’s engine — never a shell command. This is a hard rule, not an open question: a committed manifest must not be able to execute arbitrary commands on a teammate’s machine on first arch_query. Two kinds in v1:

  • path — a glob over the tree (an anchor set of files/dirs).
  • symbol — a tree-sitter symbol query (e.g. class GraphReducer, interface I*Repository) scoped by an in: glob.

A structural (ast-grep-style) locator and an LSP-resolved locator are future kinds, gated on the sandbox/host-dependency decisions in §12.

“User #0” — this repo, hand-authored

Honesty fix (from the generality review). The manifest format is language-agnostic — it’s just name → locator → anchors. But these five concepts are irreducibly Angular/TS/this-repo-specific and were written by a human who knows the repo, not produced by a general bootstrap. On a fresh unknown repo, auto-proposal yields generic “most-imported modules” hubs — not named architecture concepts. So: rich vocabulary requires human authoring; auto-bootstrap is a cold-start convenience, deferred. We validate the wedge by hand first.

Below is the verified v1 manifest for this repo, rewritten into the engine-resolved locator model (the research draft used shell grep/find; v1 does not). Paths verified against the current tree.

# .arcscope/vocab.yaml — architecture vocabulary for "user #0" (this repo)
# Locators resolve through arcscope's engine. No shell-out. Kinds: path | symbol.

concepts:
  repository-tokens:
    title: Repository-token pattern (I{Name}Repository)
    description: DI token + interface + Firestore impl per data-access contract; inject the token, never the class.
    locators:
      - { kind: symbol, query: "interface I*Repository", in: "libs/data-access/src/lib/interfaces/**" }
      - { kind: symbol, query: "const *_REPOSITORY = InjectionToken", in: "libs/data-access/src/lib/tokens/injection-tokens.ts" }
      - { kind: path,   glob: "apps/**/core/repositories/firestore-*.repository.ts" }

  editor-state-flow:
    title: "Editor state pipeline: tool → facade → router → reducer → state"
    description: Unidirectional dispatch for every editor mutation.
    stages:                       # ordered → arch_query returns the flow in sequence
      - { title: GraphEditorFacade,       kind: symbol, query: "class GraphEditorFacade",       in: "libs/features/graph/**/services/core/**" }
      - { title: ActionRouterService,     kind: symbol, query: "class ActionRouterService",     in: "libs/features/graph/**/services/core/**" }
      - { title: GraphReducer,            kind: symbol, query: "class GraphReducer",            in: "libs/features/graph/**/state/**" }
      - { title: GraphEditorStateService, kind: symbol, query: "class GraphEditorStateService", in: "libs/features/graph/**/services/core/**" }

  shape-plugin-registry:
    title: Shape plugin registry & registration
    locators:
      - { kind: symbol, query: "class PluginRegistryService", in: "libs/features/graph/**" }
      - { kind: symbol, query: "const CORE_PLUGIN_PROVIDERS", in: "libs/features/graph/**/plugins/**" }
      - { kind: path,   glob: "libs/features/graph/**/plugins/*.plugin.ts" }

  schema-migrations:
    title: Document & element schemaVersion migrations
    locators:
      - { kind: symbol, query: "class DocumentNormalizerService", in: "libs/features/graph/**/persistence/**" }
      - { kind: symbol, query: "method normalizeElement",          in: "libs/features/graph/**/plugins/**" }

  firestore-import-boundary:
    title: "@angular/fire/firestore import perimeter"
    # DEGRADED in v1: we anchor the allowed island + point at the rule. Live-extracting the
    # allowlist out of eslint.config.mjs needs a 'custom' locator (parse a JS config object) —
    # deferred until the locator host/sandbox model is settled (see Open decisions).
    locators:
      - { kind: path, glob: "apps/**/core/persistence/**" }   # the only place firestore imports are allowed
      - { kind: path, glob: "eslint.config.mjs" }             # where the rule is defined

The generality review caught two things baked in above: firestore-import-boundary is genuinely config-introspection, not code navigation — so it ships degraded (anchors only) with an honest note, rather than pretending a v1 locator can parse an ESLint flat-config object. And authoring this by hand surfaced exactly how repo-specific these predicates are — which is the evidence that reliable auto-bootstrap is hard.

Drift detection — anchor-relative, not magnitude-relative

The dangerous staleness is not “resolved to zero” — it’s “still resolves to a plausible-but-wrong set after a refactor.” So at accept time we store the resolved set (content-hashed anchors); arch_query flags drifted when the current resolution diverges from the stored anchors, not only when the count collapses. That’s what actually delivers “drift is a signal, not a surprise.”

8Cold-start, setup & adoption

A real user starts from nothing: an unknown local checkout, no package, no server. So cold-start, setup, registration, and adoption are first-class product surfaces — designed, not assumed.

Cold-start & the index

One command, npx arcscope init, run in the repo root: detect languages by extension, lazy-load the needed WASM grammars, run a first index pass, write registration config, and update .gitignore. No persistent SQLite store or file watcher in the slice (a review cut — see §11): index in-memory on server start, re-index lazily on a tool call when a file’s content hash changed. We add a persistent store + incremental watcher only after a benchmark on this repo (~1,067 TS files) shows on-demand re-index is too slow. We quote no performance number until that benchmark exists.

Setup & registration (Claude Code first)

init writes a project-scoped .mcp.json at the repo root — the committed, git-shared mechanism Claude Code reads (equivalent CLI: claude mcp add --scope project --transport stdio arcscope -- …). First connection prompts for approval (stdio servers are a trust boundary). A single bin branches on argv (init | serve). Other clients (Cursor/Cline/Codex, Claude Desktop) use the same mcpServers shape; init prints the snippet.

Honesty fix (feasibility review): “no network” includes server spawn. A committed command: npx arcscope serve makes every session start an npx resolution that can hit the network — violating the local-only guarantee at spawn time and risking an unpinned breaking version mid-project. So for the dogfood/committed config we point at a locally-resolved binary (prefer node_modules/.bin when arcscope is a devDependency, or a pinned version), so server start is offline and deterministic.

Adoption — the #1 risk, treated empirically

A registered tool the agent ignores is worthless, and the product’s single success criterion is “the agent reaches for arcscope instead of grep.” The levers are real but all prompt-engineering: the MCP server instructions field (the Tool-Search gateway loaded at session start), “Use when …” tool descriptions, and a short CLAUDE.md preference nudge (project rules take precedence over MCP discovery).

Honesty fix (feasibility review). (1) This repo’s CLAUDE.md has no existing navigation guidance to amend — the nudge is net-new (verified). (2) More importantly, adoption is a bet, not a given, so Phase 0 treats it as an empirical gate with a kill-criterion: run a paste-able test prompt against the real agent across N realistic navigation tasks in this repo and measure whether it invokes the tool unprompted. If it won’t, the wedge is dead regardless of engine quality — fix instructions/descriptions/nudge before building anything else. A local-only invocation counter (writes to .arcscope/, never the network) quantifies the grep-vs-tool ratio during dogfooding.

9Distribution & the dogfood loop

arcscope is a standalone repo from day one — not a package inside this Nx monorepo. This repo is only its first consumer. (That sidesteps the whole “uninstallable @graph-knowledge/* workspace-dep tarball” class of bug the synthesis worried about — there are no such deps.)

Public npm publish is a milestone, not a test step: npm versions are permanently immutable (a burned version is burned forever; unpublish is heavily restricted), so iterating via public publishes litters history with dead releases. The ladder:

  • Inner loop (this repo): run the local bin directly — npm link or an absolute/node_modules/.bin path in .mcp.json. You don’t need a registry to consume your own server in your own repo.
  • Stage 0 — npm pack --dry-run: lock the file manifest (catch a missing dist/ or the bin pointing outside files).
  • Stage 1 — npm pack + install the tarball into a scratch consumer: exercise both faces — arcscope init from the installed shim and npx arcscope serve launching the stdio server.
  • Verdaccio — deferred to immediately before the first public publish (a review cut: rehearsing the publish round-trip has no value until you intend to publish). When we get there, it lets us re-publish the same 0.0.1 freely without burning public coordinates.
  • Milestone — first public publish under --tag next (so latest stays clean; testers opt in via @next), only once the slice works end-to-end. Cut 0.1.0 on latest later.

Packaging shape (mirrors the official MCP servers): "type":"module", single "bin" branching on argv, "files":["dist"] (+ vendored .wasm grammars & tags.scm), "engines":{"node":">=20"}, shebang on line 1, tsc && shx chmod +x. Log to stderr only — a stray console.log on stdout corrupts the JSON-RPC stream and silently hangs the server.

10Roadmap

Each phase ships value on this repo and ends with a measurement, not a hope. Phases 0–2 are the committed plan; everything past is explicitly out of v1.

Phase 0 Spine + the adoption gate  (TS/JS only)

  • web-tree-sitter engine, lazy grammar load, tags.scm definition extraction → in-memory index (no persistent store, no watcher).
  • One tool: find_def — definitions genuinely beat grep with no resolution machinery.
  • arcscope init / serve bin; .mcp.json points at the local resolved bin; logging to stderr.
  • Adoption gate (do this early): run the test prompt against the real agent; measure unprompted invocation; kill-criterion if it won’t bite.
  • Acceptance: measure WASM cold-index time on this repo before quoting any figure.
  • Ships: Pain 1 (definitions) end-to-end — and the load-bearing adoption bet is validated before further investment.

Phase 1 References + dependency graph

  • Build the import-edge index (resolving tsconfig paths + barrel re-exports — the cheap 80%). This is shared machinery for both tools below.
  • find_refs — now actually beats grep because it follows aliases/barrels; cross-module hits it can’t resolve are honestly labeled “unresolved candidate”, never dressed up as precise.
  • dep_graph — file→file import graph; directory grouping primary, tsconfig-paths where present. PageRank ranking deferred until neighborhood queries prove noisy without it.
  • Worker-pool parallelism iff the P0 cold-index benchmark was too slow single-threaded.
  • Ships: Pain 1 (fully) + Pain 3.

Phase 2 The differentiator — architecture vocabulary

  • .arcscope/vocab.yaml schema + loader; one engine-resolved locator path (symbol query + path glob), no shell-out.
  • arch_list + arch_query, answered live; anchor-relative drift detection.
  • Hand-author this repo’s vocabulary (the §7 manifest) and validate the wedge against Serena-style static prose. Auto-bootstrap stays deferred.
  • First public npm publish (--tag next) around here, once solid.
  • Ships: Pains 2 & 4 — and the unique, defensible wedge is live.

Later / explicitly out of v1

Optional LSP precision overlay (tree-sitter fallback + tier labeling) · auto-bootstrap (PageRank hubs + a few cross-language heuristics, validated on a non-JS repo before claiming value) · persistent SQLite + incremental watcher (only if benchmarks demand) · more grammars/languages · ast-grep as an optional accelerator (never required) · CI conformance mode for drift · SCIP / stack-graphs as far-future precision substrates · public GA (0.1.0/1.0.0 on latest).

11What we cut, and why

The synthesized design was, per the scope review, ~2× the wedge. These cuts come straight from the three review lenses and are the difference between this spec and the raw synthesis.

Cut / deferredWhy
LSP / ctags / SCIP / stack-graphs from the committed planThe spec’s own logic says “wrap-an-LSP is commodity → differentiation can’t live there.” It’s also the most operationally fragile piece. tree-sitter-only for v1; precision tier is a label with room to grow.
find_symbol+find_refs both in P0 → find_def in P0, find_refs in P1Definitions are the easy half (real grep win, no resolution). References only beat grep after the import-edge index exists — and on this repo’s barrel/alias-heavy code, an unresolved tree-sitter “ref” is the same false-positive soup as grep, mislabeled as precise.
Auto-bootstrap of the manifest (PageRank + pattern heuristics)Highest-effort, lowest-validated, and reverse-engineered from this repo’s Angular/Redux idioms — they don’t fire on a Go/Rust/Python repo. Validate the wedge with a hand-written manifest first.

12Open decisions — need your call

  1. Name — decided: arcscope (free on npm as of 2026-06-19). Still open: scoped (@you/arcscope) vs unscoped — scoped publishes restricted-by-default, so needs --access public on first publish.
  2. Grammar delivery. Bundle a curated, pinned set of .wasm grammars + vetted tags.scm (predictable, offline, but heavier package) vs locate them from node_modules (lighter, but version-drift & the no-network guarantee fight). Recommendation: bundle a small pinned set (TS/JS for the slice).
  3. Persistence driver (when we add it). better-sqlite3 (native, fast, but breaks a literal “no native build”) vs node:sqlite (no native dep, but experimental on the Node-20 floor). Resolve by restating the principle as “no per-language native build” and accepting one native store dep — when we need a store.
  4. Nx grouping signal. Your standing directive is general-first, so we do not read nx.json/project.json. The honest cost: arcscope under-groups the Nx/Bazel monorepos that most need grouping (its only current user included), falling back to tsconfig-paths + directory. Confirm you’re OK paying that for generality.
  5. Future structural/“custom” locators & sandboxing. The firestore-import-boundary concept wants to parse a JS config object live; that implies a custom-code locator, which reopens “can a committed manifest run code on a teammate’s machine?” v1 answer: no — engine-resolved locators only. Decide later whether a sandboxed custom locator is worth it.
  6. Scope of the live answer. v1 is strictly read-only navigation. A CI “conformance mode” (fail the build when a concept drifts) is a plausible second product — confirm it stays out of v1 so it doesn’t bend the tool shapes.

13Success & kill criteria

Success

  • In this repo, the agent searches for and invokes arcscope tools instead of grep on realistic navigation tasks — confirmed via the test prompt without naming the tool.
  • find_def (P0) / find_refs (P1) return correct results for known symbols, with the precision tier labeled honestly.
  • npx arcscope init on a fresh clone produces a working .mcp.json and a non-empty index with zero manual setup and zero network calls (verifiable by monitoring the process — including at server spawn).
  • (P2) arch_query on each user-#0 concept returns live, correct locations and flips to drifted when the underlying code/anchors change.
  • The package installs cleanly from a tarball into an external scratch consumer; both bin faces run.
  • Cold index of this repo completes in a duration we have measured (and that doesn’t make first-session feel slower than grep); single-file re-index is sub-second.

Kill criteria

  • Adoption (P0): if the agent won’t invoke the tools unprompted after the instructions/descriptions/nudge are tuned, the wedge is dead — stop and rethink before building P1+.
  • Precision (P1): if import-resolved find_refs still can’t beat grep on this repo’s barrel/alias patterns, the “precise nav” half of the pitch falls to the deferred LSP tier — re-decide whether that’s acceptable for v1.
  • Wedge (P2): if a hand-authored, live-recomputed manifest demonstrably doesn’t beat static prose in practice, the differentiator is unproven — don’t build auto-bootstrap on top of it.
